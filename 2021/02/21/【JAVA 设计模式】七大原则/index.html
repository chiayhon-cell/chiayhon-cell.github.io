【设计模式】七大原则

发表于 2021-02-21 更新于 2021-03-01
本文字数： 687

前言

设计模式的目的

设计模式的核心是让代码具有可扩展性。可扩展性又可以分为

  1. 自己扩展自己

  2. 第三方扩展。

扩展性的根基：可读性

没有可读性的代码，可扩展性无从谈起。可读性包括代码排版、变量命名规范、代码结构和恰当的设计模式的运用。为什么大家这么讨厌别人的遗留下来的项目，十之八九是可读性太差，扩展性没有，改一处而动全身。有时候我们经常要在性能和可读性上做抉择，请尽可能选择可读性。性能上差别一般情况不会太大，特别是前端。

对框架而言，需要同时考虑自己扩展自己和第三方扩展。而对于普通项目来说，更多的是考虑自己扩展自己。不管谁扩展谁，关键是如何运用设计模式。

七大设计原则

¶单一职责(如何定义类)

总结：一个类只干一件事情

实现：将 单个多职责类 分成 多个单职责类

目的：便于理解，减小粒度，提高可读性

¶接口隔离原则(如何定义接口)

总结：一个接口只干一件事情

实现：将 单个多职责接口 分为 多个单职责接口

目的：功能解构，高内聚低耦合，避免冗余扩展

¶依赖倒置原则(如何使用接口)

总结：高层不依赖低层且两者应依赖抽象，抽象不依赖细节且细节应依赖抽象

实现：抽象定义依赖关系，具体定义业务逻辑。高层的具体依赖抽象低层的抽象。

目的：面向接口编程，利于代码的升级与扩展

¶里氏替换原则(如何继承)

总结：继承避免重写，若要重写，也不应影响父类方法的含义。

实现：子类不能完整实现父类业务，或父类的方法在子类中不能满足或适用，则断绝继承关系，以依赖，聚集，组合关系替代

目的：类间解耦，防止继承泛滥

¶开闭原则

总结：对扩展开发，对修改关闭

实现：抽象约束，封装变化

目的：减少维护代码时带来的风险
